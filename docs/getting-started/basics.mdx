# Getting Started: Basics

**VirtualMetric Director** telemetry processes are configured with YAML format files. In this chapter, we will give you a brief summary of how these files are used and the minimum configurations required.

## The Files

We recommend that you read [this section](/docs/configuration/index.mdx#files) first to have a basic understanding of what type of files are used, where they are placed, and how they can be arranged.

For our purposes here, we will put the files in the examples under the `config` directory of `<vm_root>`, i.e. the directory where you have placed **Director**.

## The CLI

You can invoke the processes you want **Director** to run from a terminal. The `-help` switch will enlist the available parameters.

Here are a few typical use cases:

* Collecting _Syslog_:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>
  </Tabs>

* Viewing logs with a specific debug level:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>
  </Tabs>

* Running a specific pipeline with validation:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>
  </Tabs>

We will provide the relevant commands to execute the configured examples.

## The Components

To stream and collect data points, to process them, and to forward them to destinations, **Director** uses the following types of configurable components:

* **Devices** - Data sources that collect or receive data. See the **Configuration** heading [here](/docs/configuration/devices/index.mdx#configuration).

* **Targets** - Destinations where processed data is sent. See the **Configuration** heading [here](/docs/configuration/targets/index.mdx#configuration).

* **Pipelines** - String of processors that transform the data. See the **Configuration** heading [here](/docs/configuration/pipelines/index.mdx#configuration).

* **Routes** - Predefined data flows from devices through pipelines to targets. See the **Configuration** heading [here](/docs/configuration/routes/index.mdx#configuration).

After reviewing the indicated sections, you can try the examples to see how **Director** works.
