# Getting Started: Basics

**VirtualMetric Director** telemetry processes are configured with YAML format files. In this chapter, we will give you a brief summary of how these files are used and the minimum configurations required.

## The Files

We recommend that you read the [Configuration: Overview](/docs/configuration/index.mdx) first to garner a basic understanding of placement of files and the syntax used to edit the configuration files.

For our purposes here, we will place our files under the `config` directory of `<vm_root>`, i.e. the directory where you have placed **Director**.

## The CLI

You can invoke the processes you want **Director** to run from a terminal. The `-help` switch will enlist the available parameters.

Here are a few typical use cases:

* Collecting _Syslog_:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=generator -protocol=syslog -address=127.0.0.1:514
      ```
    </TabItem>
  </Tabs>

* Viewing logs with a specific debug level:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=console -debugLevel=3
      ```
    </TabItem>
  </Tabs>

* Running a specific pipeline with validation:

  <Tabs groupId="os-options">
    <TabItem value="windows" label="Windows" default>
      ```powershell
      vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>
    
    <TabItem value="linux" label="Linux" default>
      ```bash
        vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>

    <TabItem value="macos" label="macOS" default>
      ```bash
        vmetric-director -mode=pipeline -name=<pipeline_name> -validate
      ```
    </TabItem>
  </Tabs>

In the following sections, we will provide the relevant commands to execute the configured examples.

## The Components

To stream and collect data points, to process them, and to forward them to destinations, **Director** uses the following types of configurable components:

* **Devices** - Data sources that collect or receive data. You can find their basic configuration [**here**](/docs/configuration/devices/index.mdx#configuration).

* **Targets** - Destinations where processed data is sent. You can find their basic configuration [**here**](/docs/configuration/targets/index.mdx#configuration).

* **Pipelines** - String of processors that transform the data. You can find their basic configuration [**here**](/docs/configuration/pipelines/index.mdx#configuration).

* **Routes** - Predefine data flows from devices through pipelines to targets. You can find their basic configuration [**here**](/docs/configuration/routes/index.mdx#configuration).

After reviewing s of the indicated sections, you can try the examples to see how **Director** works.
